01 Overview

What This Project Demonstrates

This project demonstrates a fully serverless, event-driven document processing pipeline built on AWS. When a PDF is uploaded to an S3 bucket, the pipeline automatically extracts all text from the document using Amazon Textract and stores the results in DynamoDB, with zero servers to manage, zero polling loops, and near-zero idle cost.

This architecture reflects real-world patterns used in document intake systems, insurance claim processing, healthcare form digitization, and legal document management.

02 Architecture

How the Pipeline Works

The pipeline is fully event-driven — nothing runs unless something happens. A PDF upload to S3 fires a PUT event that invokes Lambda #1, which starts an asynchronous Textract job and passes the TextractJobComplete SNS topic as its notification destination. When Textract finishes, it publishes a completion message to SNS, which triggers Lambda #2. Lambda #2 paginates through all Textract result blocks using NextToken, then writes the extracted text, Job ID, and timestamp to DynamoDB. Both Lambda functions emit structured logs to dedicated CloudWatch log groups, capturing execution details, Textract Job IDs, extracted line counts, and any errors — providing full observability across the pipeline with zero additional infrastructure.

Step 01
PDF Lands in S3
A PDF is uploaded to the S3 bucket, triggering an s3:ObjectCreated:Put event.
Step 02
Lambda #1 Triggers
S3 invokes Lambda #1, which starts an async Textract job, passing the SNS topic as the notification destination.
Step 03
Textract Processes
Amazon Textract performs asynchronous OCR on the document using StartDocumentTextDetection.
Step 04
SNS Publishes Completion
When the job finishes, Textract publishes a completion message to the TextractJobComplete SNS topic.
Step 05
Lambda #2 Retrieves Results
SNS triggers Lambda #2, which calls GetDocumentTextDetection and paginates through all result blocks.
Step 06
Stored in DynamoDB
Extracted text, Job ID, and timestamp are written to the ClientIntakeForms DynamoDB table.

04 Best Practices Applied

Architecture Decisions

Asynchronous Processing
Using StartDocumentTextDetection instead of DetectDocumentText means the pipeline handles multi-page documents and does not time out on large files.
Event-Driven Design
Nothing in this pipeline polls or waits. Every component reacts to an event, which keeps costs near zero when idle.
Least-Privilege IAM
Each Lambda has its own role with only the permissions it needs. No role has admin access or wildcard resource permissions beyond what is necessary.
Pagination Handling
Lambda #2 loops through all Textract result pages using NextToken, ensuring no text is missed on long documents.
Environment Variables
All ARNs and resource names are stored as environment variables, not hardcoded, making the project portable across environments.
Single Responsibility
Every component has a single responsibility. Every trigger is event-driven. Nothing runs unless something happens.
